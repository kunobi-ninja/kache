Thanks so much for raising this, and for the detail on your layout — the symlinked build-forest setup is exactly the kind of real-world configuration worth pressure-testing against, so this is genuinely useful.

You've read the blog sentence exactly as written, and the concern is spot-on *for what that sentence describes*. The post simplified things to keep the mental model easy to follow: it describes only the happy path, where the common ancestor of the build dir and the source file is the repo root. The actual implementation is more defensive than that one line implies, and I think it already covers your case — let me point at the specifics:

- **Separate source/build sentinels already exist.** When the common ancestor *isn't* a useful shared root, the code falls back to two distinct maps — `cwd → <CC_BUILD>` and the source dir → `<CC_SOURCE>` — which is precisely the "separate sentinels for source and build tree" you're asking for. (`cc_prefix_maps_for` in `src/compiler/cc.rs`, the `else` branch.)
- **There's a guard against the `$HOME`-becomes-the-prefix degeneracy you describe.** The single common-root sentinel is only used when the ancestor is genuinely meaningful: `stable_cc_common_root` requires the common path to have ≥3 normal components (and rejects the tempdir). For a sibling/out-of-tree build whose ancestor with the source resolves up around `$HOME`, that check fails and it drops to the split `<CC_SOURCE>` / `<CC_BUILD>` sentinels — so the replacement doesn't degenerate to "strip my home dir and gain nothing."
- **Symlinks are accounted for** — the ancestor is computed both on the literal absolute paths and on `canonicalize`d paths, which matters for your `builds → $buildsforest/$project` symlink shunting.
- **Generated sources** that live in the objdir get the repo root folded back in via the `-I` include dirs, so cross-variant generated TUs still normalize.
- **Explicit override:** if the auto-derivation ever picks a root narrower than you'd like, `KACHE_BASE_DIR` (the analog of ccache's `CCACHE_BASEDIR`) lets you name the prefix to strip directly → `<CC_BASE>`.

So the short version: the design already does what you're proposing as the general-case fix — it just isn't visible in the blog's one-liner.
