# kache 0.6.0 — a shareable Rust + C/C++ build cache

kache is a content-addressed Rust build cache that plugs in as a `RUSTC_WRAPPER`.
0.5.0 designed a correct cache key; 0.6.0 makes that key **portable across machines and
checkouts** and extends caching to the **C/C++ that crates compile in build scripts**.

## Why

A shared cache only helps if two checkouts of the same source hit the same entry — and
they usually don't, because absolute build paths leak into the key. 0.6.0 closes that gap,
and pulls the C/C++ half of many build times (`ring`, `openssl-sys`, Firefox) under the
same cache.

## What's new

- **Cross-checkout sharing.** `KACHE_BASE_DIR` normalizes the build root; opt-in
  `[cache] path_only_env_vars` (or `KACHE_PATH_ONLY_ENV_VARS`) marks build-location env
  vars so they normalize instead of poisoning the key.
- **C/C++ via clang-cl (incl. Windows/MSVC).** kache now caches MSVC-style `cl`/clang-cl,
  not just gcc/clang — no extra config if kache is already your wrapper.
- **`KACHE_LOCAL_ONLY`** — disable all remote I/O for air-gapped/private builds.
- **`KACHE_VERIFY_RESTORES`** — opt-in re-hash of restored blobs before use.

## What we learned (the Rust-specific parts)

- **`env!()` bakes absolute paths into rlibs.** Firefox's `mozbuild.rlib` embedded
  `TOPOBJDIR`/`TOPSRCDIR`, so the *artifact* differed per checkout — no key could fix it.
  A correct key isn't enough when the output embeds the path.
- **A degraded dep-info pass weakens the key** — so we refuse to cache from one.
- **Racy compiles** (inputs changing mid-build) must not be stored; opt-in guard added.
- **clang-cl is a dialect, not a flag** — needed MSVC flag tables and a `/EP` preprocess
  probe, not gcc/clang's `-E`.

Hardening for shared use: length-prefixed key fields (cache `v16`), file metadata folded
into the dedup hash, and a cross-process `gc.lock`.
